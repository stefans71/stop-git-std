What Should I Do?

Step 1: Stop using it immediately

Do not run Paperclip until you have completed the steps below. If it is currently running, shut it down. If it is running as a service or in Docker, stop the container.

Step 2: Check if your credentials were exposed

Open a terminal and run these commands to check for leaked login tokens in the log files:

# Check if the hidden folder exists and how big it is
ls -lah ~/.paperclip/

# Search log files for leaked login tokens
grep -ri "bearer\|authorization\|token\|session" ~/.paperclip/logs/ 2>/dev/null | head -20

# Check the size of log files (one user had 816 MB)
du -sh ~/.paperclip/logs/ 2>/dev/null

If you see any output from the grep command, your login tokens were written to disk in readable text. Anyone or any program with access to your computer could have read them.

Step 3: Change your passwords and API keys

If Paperclip had access to any of the following, change them now — assume they were compromised:

  • Any API keys you entered into Paperclip (OpenAI, Anthropic, etc.)
  • Any passwords or tokens stored in Paperclip's settings
  • Your Paperclip account password (if you used one elsewhere, change it there too)
  • Any service accounts or OAuth tokens connected to Paperclip

Also check your AI provider billing dashboards for unexpected charges — one user reported 120 million tokens consumed in a single day.

Step 4: Update or remove

Option A — Remove it entirely (recommended):

# Delete the hidden data folder
rm -rf ~/.paperclip/

# If you cloned the repo, delete it
rm -rf /path/to/your/paperclip/

# If installed via npm
npm uninstall -g paperclipai

Option B — Update to get the security fix:

The "patched version" v2026.410.0 referenced in the security advisory does not exist as a downloadable release. The only way to get the fix is to pull the latest code from the master branch:

# If you have an existing clone, pull the latest
cd /path/to/your/paperclip/
git pull origin master

# If you need a fresh copy
git clone https://github.com/paperclipai/paperclip.git

Warning: Even after updating, Paperclip still defaults to giving AI agents full access to your files without asking (dangerouslySkipPermissions = true), still has no code review process, and still has open credential logging issues. Updating fixes the remote code execution bug but not the underlying security culture.

Step 5: Scan your system

If you were running Paperclip during the vulnerability window (March 2 – April 10), an attacker could have run commands on your computer. Check for signs of compromise:

# Check for unfamiliar processes
ps aux | grep -v "$(whoami)" | head -20

# Check for recently modified files in system directories
find /usr/local/bin /usr/bin -mtime -30 -type f 2>/dev/null | head -20

# Check for unfamiliar cron jobs
crontab -l 2>/dev/null
ls -la /etc/cron.d/ 2>/dev/null

# Check for unfamiliar SSH keys
cat ~/.ssh/authorized_keys 2>/dev/null

If you find anything unfamiliar, assume your system was compromised. Back up your data and consider a clean reinstall of your operating system.

Step 6: Use this prompt to get help from an AI assistant

If you're not comfortable running the commands above, copy and paste this prompt into ChatGPT, Claude, or any AI assistant. It will walk you through the cleanup step by step:

I installed software called Paperclip (paperclipai/paperclip from GitHub) and I just learned it had serious security vulnerabilities. I need your help to:

1. Check if I have the hidden folder ~/.paperclip/ on my computer and how big it is
2. Search the log files in that folder for leaked login tokens (Bearer tokens, session tokens, authorization headers)
3. Tell me what API keys or passwords I need to change based on what Paperclip had access to
4. Help me either remove Paperclip completely or update it to get the security fix
5. Check my system for signs that someone may have accessed my computer through the vulnerability
6. Check my AI provider billing (OpenAI, Anthropic, etc.) for unexpected charges

I am not a technical person. Please give me exact commands to run and explain what each one does before I run it. Ask me what operating system I am using first (Mac, Windows, or Linux).

The specific vulnerabilities were:
- Anyone on the internet could run commands on my computer without logging in (fixed April 10, but the patched release was never published)

Security fix hidden from changelogs so you would never know

The bug described above — where anyone on the internet could run commands on your computer without logging in — was fixed, but the release containing that fix (v2026.410.0) was labeled "chore:" instead of "fix:" or "security:". This label tells automated changelog tools "this is just routine housekeeping" — so the security fix never appears in release notes. Users who check changelogs to decide when to update would never see it.

It gets worse: as of today (April 15 — 11 days after the fix), version v2026.410.0 is still not listed on the GitHub Releases page. We verified this just now. The last visible release is v2026.403.0 from April 4 — six days before the security fix. This means two different things depending on how you install Paperclip:

If you clone the master branch (git clone) — you do get the fix, because the commit is in the git history.
If you install from a release, tag, or package manager — you are running vulnerable code. The latest release is v2026.403.0 from April 4, six days before the fix. The version the advisory told users to install (v2026.410.0) was never published — it does not exist as a tag, release, or downloadable package. We verified this today (April 15, 11 days after the advisory). This is worse than not publishing an advisory at all. Here's why:

The advisory tells users "upgrade to v2026.410.0 to be safe." But that version does not exist. If you try to install it, you get an error. You can't follow the instructions even if you find them.

Most users will never see the advisory. GitHub does not send a notification to everyone who cloned a repo. The only people who get automatic alerts are those using Paperclip as a dependency in their own project with Dependabot enabled. If you just downloaded the repo, cloned it, or run it from Docker — nobody tells you.

"Can't I just update?" If you originally cloned the repo from the command line, yes — running git pull would pull in the fix from the master branch. But the advisory doesn't say that. It says "install v2026.410.0" — which doesn't exist. And if you installed from a packaged release or version tag (which is how most people install software), there is no update available.

Meanwhile, the security community thinks this is fixed. Security databases and vulnerability scanners see a "patched version" listed in the advisory and mark the issue as resolved. Automated tools stop flagging it. The result: everyone moves on, but the actual fix was never shipped to users. A ghost patch that creates a false sense of resolution.
