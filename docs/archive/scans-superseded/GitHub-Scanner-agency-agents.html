Verdict

Caution — review evidence is weak and distribution is not pinned

No direct install-time remote fetch logic was found in the repo shell scripts. The concern is that a local checkout can populate Claude Code, Copilot, Cursor, Windsurf, Aider, OpenClaw, Qwen, Kimi, and Gemini-adjacent locations with prompt content while recent merged PRs in the public sample had no review events.

What We Found

Warning Ongoing

F-review — Recent merged PR sample shows no independent review before install-affecting changes ship

The 20 most recent merged PRs visible through the public API all returned zero review events, including install-path and security-policy changes.

This matters because the repository distributes prompt content into agent and rules directories. A bad change can ship with little upstream friction if review remains this thin.

  • GitHub search reported 119 merged PRs total.
  • Sampled the 20 most recent merged PRs from the public pulls API; all 20 had zero review events via the pull reviews endpoint.
  • PR 437 was authored and merged by the owner and had zero review events.
  • PR 323 and PR 410 were merged by the owner and also had zero review events.
If you install: pin the exact commit you trust and inspect install-related diffs before updating. If you maintain: require one reviewer for scripts, workflows, SECURITY.md, and integration outputs.
Warning Active

F-distribution — The documented install path relies on a local checkout rather than a tagged or signed artifact

README guidance tells users to run convert.sh and install.sh from a checkout. The repository had zero tags and zero releases at scan time.

The next user who clones or pulls current main gets the newest prompt payload. That risk is amplified because the payload is agent content that several tools auto-load or surface prominently.

  • README Quick Start uses local scripts from the checkout and does not pin a release or tag.
  • GitHub releases endpoint returned an empty list.
  • Git tags list was empty.
  • Installer destinations include user-wide and project-scoped agent or rules paths such as ~/.claude/agents, ~/.copilot/agents, .cursor/rules, CONVENTIONS.md, .windsurfrules, ~/.config/kimi/agents, and ~/.openclaw/agency-agents.
If you install: use a specific commit SHA and keep a trusted local snapshot. If you maintain: publish tagged releases or signed bundles and document a pinned install path.
Informational Informational

F-surface — The repository has low classic code-execution surface but high prompt-distribution surface

Static checks did not show install-time network fetches or committed secret files in the shipped shell scripts. The main payload is 184 Markdown prompt files copied into tool-specific locations.

This reduces traditional malware-style risk, but it shifts trust to prompt quality, tool auto-load behavior, and maintainer controls.

  • No root dependency manifests were found for npm, PyPI, Cargo, Go, Ruby, Maven, or Composer.
  • No .env, PEM, SSH key, or credentials filenames were found.
  • The only workflow is lint-agents.yml; it uses pull_request, not pull_request_target.
  • Grep hits for fetch and subprocess patterns appeared mainly inside Markdown prompt examples, not inside repository-executed shell code.
Review only the specific agents and integrations you plan to install, with extra care for always-on or user-wide destinations.

OSSF Scorecard

OSSF Scorecard is an OpenSSF service that summarizes repository hygiene checks such as branch protection, code review, token permissions, and pinned dependencies. This repo returned HTTP 404 from the public Scorecard API during the scan.

https://securityscorecards.dev/

osv.dev

osv.dev is a public package-vulnerability database. It is most useful when a repository has direct dependency manifests. This repo did not expose such manifests at the root, so there was no meaningful osv dependency set to query.

https://osv.dev/

gitleaks

gitleaks scans repositories for secrets. It was not installed in this environment, so secrets in history were not checked here.

https://github.com/gitleaks/gitleaks

GitHub API Budget

GitHub API rate limits matter because PR-history review and governance checks are request-heavy. This scan ran under the public unauthenticated budget, so deeper GraphQL-only checks were out of reach and review conclusions were explicitly sample-based.

https://docs.github.com/rest/using-the-rest-api/rate-limits-for-the-rest-api

Section 08 — How This Scan Works

Phase 1: prep and commit pinning.

Phase 2: evidence gathering from local source, public GitHub REST, Scorecard lookup, issue and PR sampling, and install-path inspection.

Phase 3: findings bundle with facts, inference, and synthesis separated.

Phase 4: canonical Markdown first, then derived HTML.

Phase 5: validator loop until both outputs pass.

Phase 6: deliver results with explicit coverage gaps instead of guessing.

This scan treated repository files, PR text, and issue text as untrusted data. It favored local grep, minimal-context reads, and explicit coverage accounting over narrative trust or guessed results.
