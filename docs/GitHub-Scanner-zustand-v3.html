What Should I Do?

✓ Safe to install today ⚠ 1 consumer-side mitigation
Install is safe — the library itself is clean. For production use, pin exact versions to protect against future upstream compromise.

Step 1: Install zustand — the library is clean

Non-technical: Zustand is a pure React state library. It has zero runtime dependencies, no install-time scripts, no shell calls, no network calls, and nothing in its source tree that resembles a dangerous primitive. The published npm package is built through GitHub Actions with OIDC authentication.

npm install zustand@5.0.12
# or: pnpm add zustand@5.0.12
# or: yarn add zustand@5.0.12

Step 2: Pin to exact version for production

Non-technical: Because there is no branch protection on main, a compromised maintainer account could push malicious code in a future release. Pinning your version protects you until you manually verify the next upgrade.

# In package.json, use exact version (no ^ or ~):
"zustand": "5.0.12"

Step 3: If you maintain this repo — enable branch protection

Non-technical: Navigate to github.com/pmndrs/zustand/settings/branches and add a branch protection rule for main requiring at least one review. Add a SECURITY.md with a private disclosure channel. Both are free and take a minute each.

Warning Active Ongoing

No Branch Protection on Default Branch (C20 Governance Single-Point-of-Failure)

Pin exact versions in production package.json. If you maintain this repo: enable branch protection at Settings → Branches requiring 1 approver.

What this means for you: Anyone with write access to pmndrs/zustand can push code directly to main without a review gate. This does not mean the code is compromised today — it means the safety net is missing.

Threat model (F13): An attacker who compromises a maintainer’s GitHub account (via phishing, leaked token, session hijack from a malicious browser extension, or compromised IDE plugin) could push directly to main. The publish.yml triggers on release events, so the attacker would also need to create a release to ship to npm. This is a two-step attack, but neither step has a review gate.

What this does NOT mean: The library code is clean — zero runtime deps, zero install hooks, no network, no dangerous primitives. This finding is about the upstream process gap, not the current code.
